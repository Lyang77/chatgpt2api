# 日志列表模型列与筛选条件持久化设计

## 目标

- 调用日志列表展示“模型”列，内容取自现有日志数据的 `detail.model`，缺失时显示 `-`。
- 隐藏列表中的“类型”列；日志类型仍由页面顶部的类型选择器控制。
- 将日志查询条件同步到 URL 查询参数，使浏览器 F5 刷新后恢复条件并自动重新查询。
- 日志详情只保留弹窗正文的一个纵向滚动区，消除长文本区域与弹窗同时滚动的问题。

## 范围

需要持久化的查询条件包括：日志类型、开始日期、结束日期、令牌名称、执行账号、状态、模型、接口、批次 ID 和简述类型。

页码、每页条数、勾选项、详情弹窗、图片预览等临时界面状态不持久化。F5 后从第 1 页、默认每页 20 条开始查询。

## 前端设计

日志页首次渲染时从 URL 查询参数读取筛选条件，无法识别或缺失的值回退到当前默认值。用户点击“查询”时，将当前筛选条件写入 URL；空值从 URL 中移除，然后以第 1 页请求日志。

日期或日志类型变化仍沿用当前自动查询行为，同时更新 URL，确保地址栏与实际结果一致。“清除筛选条件”只清空筛选输入；用户随后查询时 URL 中相应参数被移除。列表右上角的“刷新”只按当前条件重新请求，不改写或清空条件。

表格仅在调用日志下展示“模型”列，放在“执行账号”之后、“调用耗时”之前。模型内容使用现有 `getDetailText(item, "model")` 读取，无需修改后端接口或存储结构。原“类型”表头和单元格从表格移除。

日志详情继续由弹窗正文容器承担纵向滚动。“结果内容”“请求内容”“原始数据”中的 `pre` 不再设置固定最大高度或独立纵向滚动，长文本、请求图片和其他详情内容随弹窗正文整体滚动，页面上始终只有一个纵向滚动条。

## URL 契约

采用与后端筛选字段一致的参数名：`type`、`start_date`、`end_date`、`key_name`、`account_email`、`status`、`model`、`endpoint`、`batch_id`、`summary`。

URL 只表达已应用的查询条件。输入框中尚未点击“查询”的文本不会提前改变地址栏；日志类型和日期因现有交互会自动应用，因此变更后立即同步。

## 异常与兼容

- 未提供 URL 参数时保持当前默认行为：调用日志、无筛选条件。
- 不支持的日志类型回退为调用日志；状态和简述值作为普通字符串交给现有后端校验与筛选逻辑。
- 使用 `router.replace` 更新当前地址，不新增浏览器历史记录，避免每次查询都产生一条后退记录。
- 后端已经返回模型字段并支持模型筛选，本次不调整 API。

## 测试

- 单元测试 URL 参数与筛选状态之间的双向转换：完整条件、空值省略、默认值、不支持的日志类型。
- 组件/静态契约测试确认调用日志表格包含“模型”表头和模型值读取，并且不再渲染“类型”列。
- 静态契约测试确认详情正文保留外层滚动，同时三个文本 `pre` 不再包含内部限高或滚动样式。
- 运行现有 Web 测试与 TypeScript/Next.js 构建，确认日志详情、筛选和分页没有回归。

## 验收标准

1. 调用日志列表显示对应模型，缺失模型显示 `-`。
2. 列表不再显示“类型”列，顶部日志类型选择器仍可正常切换。
3. 设置任意查询条件并查询后，F5 刷新仍显示相同条件，并按相同条件请求数据。
4. 点击列表“刷新”不会改变查询条件。
5. 清除条件并重新查询后，URL 中对应筛选参数被移除。
6. 查看含长请求或响应文本的日志详情时，弹窗内只有一个纵向滚动条，文本和图片均可通过该滚动条完整查看。
