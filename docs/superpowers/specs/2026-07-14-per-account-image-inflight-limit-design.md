# 账号级生图在途上限设计

## 目标

允许管理员在号池管理中为每个账号独立设置生图最高在途数量。已有账号和新导入账号默认值均为 `3`；调度时只以账号字段为准，所有账号达到各自上限时等待槽位释放，不向上游发送额外请求。

## 数据契约

每条账号数据新增 `image_max_inflight: int`：

- 缺失、非法或小于 `1` 时规范化为 `3`。
- 导入、刷新、token 轮换和导出必须保留该字段。
- 管理 API 允许单账号更新该值，拒绝小于 `1` 的值。
- 原全局 `image_account_concurrency` 仅保留旧配置兼容，不再参与图片账号调度。

## 调度规则

图片候选账号逐条读取 `image_max_inflight`。只有账号当前 `_image_inflight` 小于自身上限时才可被选中；若有其他账号未满则继续轮转，若所有符合模型、套餐和来源条件的账号都已满，则在现有条件变量上等待释放通知。

## 管理界面

号池表格的“在途”列显示 `当前值 / 上限值`。编辑账号弹窗增加数值输入“最高在途数量”，默认展示 `3`，保存时与状态、代理和允许模型一起提交。通用配置页移除“单账号图片并发”，防止出现两个控制入口。

## 验证

后端测试覆盖默认值、导入持久化、更新校验、自定义上限选号和满载等待；前端通过类型检查与生产构建验证字段契约，并检查号池表格和编辑弹窗渲染。
