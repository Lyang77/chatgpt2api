# 单任务单请求流式多图设计

## 目标

图片任务不引入 `n` 或期望数量字段。一个 `taskId` 只启动一个任务线程，并且只发起一次上游流式请求。上游实际返回的所有图片都保存到该任务的 `data[]` 中；程序依据上游终态而不是图片数量判断任务完成。

## 范围

- 覆盖 `/api/image-tasks/generations` 和 `/api/image-tasks/edits` 两条任务链路。
- 覆盖普通 ChatGPT 图片链路和 Codex 图片链路。
- 任务轮询接口在任务运行期间返回已经取得的 `data[]`。
- 日志详情直接展示实际返回图片数量。
- 保留 `/v1/images/generations` 和 `/v1/images/edits` 的 OpenAI 兼容 `n` 语义；任务链路不传递、不复用该字段。

## 不做的事情

- 不根据 Prompt 解析或推断期望数量。
- 不为一个任务创建本地图片线程池。
- 不为了补足某个数量重复发起上游请求。
- 不因上游返回多张图片而创建多个 `taskId`。

## 请求与执行模型

任务接口保持现有请求结构，不增加数量字段。数量需求只由用户写在 Prompt 中，例如“生成 20 张不同构图的商品图”。

任务服务创建一个后台线程。该线程调用一次图片协议处理器；任务链路不再显式写入 `n=1`，协议默认值仅用于保持现有 OpenAI 兼容内部类型可用。普通图片链路启动一次 ChatGPT Conversation SSE，Codex 图片链路启动一次 Responses 流式请求。

Codex 当前固定要求 `exactly one image`，需要改为要求图片工具完整执行用户请求并返回所有生成结果，但不注入结构化数量。普通图片链路保持原始 Prompt，由上游根据 Prompt 决定生成数量。

## 完成判定

完成判定不依赖图片数量。

### 普通图片链路

SSE `[DONE]` 或连接正常关闭只代表当前传输流结束。若图片由异步工具生成，程序继续查询上游 Conversation 和 Tasks 状态，直到出现下列条件之一：

- 图片工具消息进入成功终态，例如 `finished_successfully`、`finished_partial_completion` 或 `end_turn=true`；
- 上游返回明确失败或内容策略终态；
- 达到配置的图片轮询超时。

当前“发现图片 ID 且短暂稳定后即返回”的逻辑不能作为任务多图链路的最终完成条件。任务链路启用严格终态等待；OpenAI 兼容接口保持现有稳定性判定，本次不改变其完成语义。

### Codex 图片链路

持续消费 Responses 流，累计所有 `image_generation_call` 结果。`response.completed` 表示成功终态，`response.failed`、`response.incomplete` 表示失败终态；若流正常关闭但没有显式终态，已有图片按 `upstream_partial` 成功保留，没有图片则报错。

### 超时与失败

- 收到终态且至少有一张图片：任务标记 `success`。
- 收到终态但没有图片：任务标记 `error`。
- 达到超时但已经取得图片：保留图片并标记 `success`，日志记录超时结束原因。
- 达到超时且没有图片：任务标记 `error`。
- 轮询预算只读取 `config.json.image_poll_timeout_secs`；严格任务模式超时、文本回复或连接异常后不换账号重发，确保一次任务只有一次上游图片请求。

## 增量结果与轮询

协议层增加图片结果回调。每次发现新的、去重后的图片结果时，回调任务服务。任务服务在锁内把新结果追加到任务 `data[]`，更新 `actual_image_count`、`updated_at` 并持久化。

`_public_task()` 已允许运行中任务返回 `data`。因此调用方继续轮询 `/api/image-tasks` 即可获得部分结果，不需要新增浏览器长连接。重复的文件 ID、URL 或相同图片内容不得重复追加。

普通图片上游可能只在异步任务完成后一次性暴露全部文件，增量展示属于“有结果即展示”，不承诺每张图片都逐张到达。Codex 流如果逐项返回图片，则逐项写入任务。

## 任务持久化

任务记录新增以下派生字段：

- `actual_image_count`：当前 `data[]` 中去重后的图片数量；默认 `0`。
- `completion_reason`：`upstream_completed`、`upstream_partial`、`timeout_with_results` 或失败原因。

任务恢复、保存和公开响应都保留这两个字段。`actual_image_count` 必须由已保存的 `data[]` 校验，不能依赖单独计数器产生漂移。

## 日志

任务调用日志详情新增：

- `actual_image_count`：实际返回数量。
- `completion_reason`：结束原因。
- `response_image_urls`：全部去重后的结果图片地址。

日志详情的“结果”页在图片画廊上方显示“实际返回数量”，不要求用户切到原始 JSON。原始详情继续保留相同字段用于审计。任务成功但因超时结束时，同时显示简短说明。

## Web 生图页

一个任务可能返回多个 `data` 项，页面不能继续只读取 `task.data[0]`。任务同步逻辑需要把同一 `taskId` 的全部结果展开为稳定的图片项，并使用 `taskId + URL 或图片内容摘要` 生成前端 ID，避免增量轮询时因数组位置变化造成闪烁。

移除页面数量选择器，不再由页面数量驱动多个独立任务。每次提交 Prompt 只创建一个任务；数量要求由 Prompt 表达。

## 兼容性

- 已有任务请求未包含数量字段，无请求兼容问题。
- 已保存的旧任务没有 `actual_image_count` 时，读取阶段按 `len(data)` 补齐。
- `/v1/images/*` 的 `n`、并行线程池和响应结构不变。
- 任务状态仍使用 `queued`、`running`、`success`、`error`，不新增前端无法识别的状态。

## 测试策略

### 后端

- 任务提交只启动一个任务线程，且任务 payload 不包含 `n`。
- 一次上游流返回多张图片时，同一任务 `data[]` 保存全部结果。
- 运行中轮询能够读取已追加的部分结果。
- 重复流事件不会产生重复图片。
- 普通图片链路等待图片工具终态，不因第一批文件 ID 稳定而提前完成。
- Codex 链路累计多个 `image_generation_call` 并识别成功、失败终态。
- 终态有图为成功、终态无图为失败、超时有图为成功、超时无图为失败。
- 严格任务模式轮询超时后不选择第二个账号、不发起第二次上游请求。
- 日志保存正确的实际数量、结束原因和全部结果 URL。

### 前端

- 单个任务的多个 `data` 项全部展示。
- 运行中轮询新增结果时只追加新图片，不闪烁或重复。
- 日志详情直接显示实际返回数量。
- 旧任务缺少新增字段时仍可正常展示。

## 验收标准

- 一次任务只产生一个本地任务线程和一次上游图片请求。
- 上游返回一张或多张时，所有结果都属于同一个 `taskId` 并写入 `data[]`。
- 任务结束依赖上游终态或超时，不依赖预期数量和短暂稳定的图片数量。
- 运行中轮询可读取当前已有结果。
- 日志详情无需查看原始 JSON即可看到实际返回数量。
